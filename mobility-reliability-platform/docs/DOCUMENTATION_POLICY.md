# 문서 운영 정책

이 문서는 Mobility Reliability Platform의 기술 기록을 서로 다른 목적의 네 스트림으로 분리하고, 각 기록이 실제 근거와 연결되도록 하는 최상위 정책이다.

## 원칙

1. **계획, 결정, 구현 결과, 장애 사실을 섞지 않는다.** 같은 사건을 여러 관점에서 기록할 수는 있지만, 원문을 복제하지 않고 상호 링크한다.
2. **미래 시점의 보고서는 미리 설계할 수 있다.** 다만 사전에 작성한 내용은 반드시 `계획`으로 표시하며, 실제 수행·성과·참석·지출로 표현하지 않는다.
3. **실제라고 적는 문장에는 검증 가능한 근거가 있어야 한다.** 커밋, 테스트 결과, 화면, 측정값, 배포 기록 또는 확인자의 수동 확인 중 하나를 연결한다.
4. **민감정보는 문서 저장소에 남기지 않는다.** 원본 GPS 좌표, 이용자 식별정보, 인증정보, 비밀키 및 내부 접근 토큰을 기록하지 않는다.
5. **기록은 삭제보다 정정과 대체를 우선한다.** 이미 다른 문서에서 참조한 기록은 고치되 변경 이유를 남기거나 새 문서로 대체하고 이전 문서에 `superseded`를 표시한다.

## 문서 스트림

| 스트림 | 위치 | 답하는 질문 | 작성 트리거 | 포함하지 않는 것 |
| --- | --- | --- | --- | --- |
| 의사결정 기록 | [`decisions/`](./decisions/) | 왜 이 대안을 선택했는가? | 되돌리기 어렵거나 여러 컴포넌트에 영향을 주는 선택 | 진행률, 회의 참석 증빙, 릴리스 홍보문 |
| 제품 업데이트 | [`product-updates/`](./product-updates/) | 제품에서 실제로 무엇이 달라졌는가? | 사용 가능하거나 운영 가능한 변경이 검증됨 | 예정 기능, 의사결정 장문, 장애 사후분석 |
| 심각한 오류·사고 | [`incidents/`](./incidents/) | 무엇이 실패했고 어떻게 복구·예방했는가? | SEV-0/1 또는 기준을 충족한 SEV-2 | 경미한 버그 목록, 책임자 비난, 추측성 원인 |
| 사람에게 전달하는 리포트 | [`reports/fixed/`](./reports/fixed/), [`reports/human/`](./reports/human/) | 해당 회차에 무엇을 계획했고 실제로 무엇을 확인했는가? | 월 2회 고정 보고 및 요청 보고 | 근거 없는 성취, 자동 생성한 참석자·사진·지출 |
| 증거 인덱스 | [`evidence/`](./evidence/) | 주장에 연결된 검증 자료는 무엇인가? | 다른 문서에서 실제 결과를 주장함 | 개인정보·원본 GPS·비밀정보, 설명 없는 파일 투하 |

## 작성 순서

1. 중요한 기술 선택은 `decisions/`에 대안, 판단 기준, 결정과 결과를 기록한다.
2. 개발 중 생성한 화면·테스트·측정값은 `evidence/` 규칙에 따라 식별자를 부여한다.
3. 검증된 변화만 `product-updates/`에 기록하고 관련 결정과 증거를 링크한다.
4. 심각한 실패가 발생하면 `incidents/`에 즉시 타임라인을 열고 복구 후 사후분석을 완성한다.
5. 월 2회 `reports/fixed/` 사전작성본을 기준으로 계획상의 위치를 유지하고, 발행 시 실제 상태와 근거를 분리한다. 별도 요청·착수·인시던트 보고는 `reports/human/`에 작성한다.

## 상호 링크 규칙

- 식별자는 문서마다 고유해야 한다. 권장 접두사는 `ADR`, `UPD`, `INC`, `HR`, `EVD`이다.
- 사람 대상 리포트의 `실제` 항목은 최소 하나의 `EVD` 또는 검증 상태가 명시된 링크를 가져야 한다.
- 제품 업데이트는 관련 `ADR`과 `EVD`를 연결하고, 장애 수정이면 해당 `INC`도 연결한다.
- 인시던트 사후분석은 당시 배포 `UPD`, 원인이 된 `ADR`(있는 경우), 검증 `EVD`, 후속 결정 `ADR`을 연결한다.
- 증거 인덱스는 이를 사용한 `UPD`, `INC`, `HR`로 역링크한다.
- 링크할 문서가 아직 없으면 식별자를 꾸미지 말고 `TBD — 문서 생성 필요`라고 적는다.

## 상태와 표현 규칙

- 내용 수명주기 상태: `planned`, `in-progress`, `verified`, `blocked`, `superseded`, `cancelled`.
- 자동 생성·검토 전 문서는 `draft`, 자동 수집했지만 사람 검토 전인 증거는 `generated`를 사용할 수 있다. 두 값은 완료나 검증을 뜻하지 않으며 발행 시 내용 수명주기 상태와 검토자를 확정한다.
- `완료`, `개선`, `안정화`, `검증`이라는 표현은 완료 조건과 근거가 있을 때만 사용한다.
- 합성 데이터 실험은 `synthetic`, 현장 데이터 검증은 `field`, 로컬 테스트는 `local`로 명시한다.
- 추정치와 관측치는 분리하고, 숫자에는 측정 기간·모수·단위를 적는다.
- 자동 생성 문서는 사람 검토 전 `draft` 상태를 유지한다.

## 금지사항

- 수행하지 않은 업무, 회의, 배포, 실증을 실제 성과로 기록하지 않는다.
- 참석자, 참석 시각, 회의 사진, 지출, 사용자 수, 성능 수치를 자동 생성하거나 추정하지 않는다.
- 계획 보고서를 나중에 실제 기록처럼 보이도록 날짜나 표현만 바꾸지 않는다.
- 장애 원인이 확정되기 전에 추측을 사실처럼 적지 않는다.
- 한 문서를 여러 스트림에 복사해 서로 다른 진실이 생기게 하지 않는다.

## 검토 책임

문서 작성자는 사실과 링크를 채우고, 검토자는 `실제` 주장과 근거의 일치 여부를 확인한다. 1인 개발 상황에서는 작성과 검토 시점을 분리하고, 검토 일시와 결과를 문서에 남긴다.
