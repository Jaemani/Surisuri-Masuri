# 프로젝트 헌장

## 1. 문서 상태

- 상태: 승인된 착수 기준
- 확정일: 2026-07-21
- 프로젝트 기간: 2026-05-01 ~ 2026-12-31
- 외부 제품명: 미정
- 내부 개발명: Mobility Reliability Platform

## 2. 문제 정의

전동보장구의 수리·점검 이력은 기관과 수리 현장에 분산되어 있고 실제 사용량과 연결되지 않는다. 사용자는 갑작스러운 고장에 대비하기 어렵고, 수리사는 과거 맥락 없이 진단하며, 복지관은 수기·엑셀 업무와 제한된 정책 근거에 의존한다.

기존 프로젝트는 QR 기반 이력 관리가 현장에서 작동할 가능성을 확인했다. 신규 프로젝트는 상표와 코드베이스를 분리하고, 스마트폰 자체 GPS와 신뢰성 분석을 중심으로 예방점검 시스템을 새로 구축한다.

## 3. 제품 원칙

1. 복지관은 초기 도입과 운영의 허브다.
2. 사용자의 앱 입력을 최소화하고 수리사의 전문 판단을 보존한다.
3. 외부 IoT 센서 없이 모바일 기기 자체 GPS를 1차 데이터원으로 사용한다.
4. 연결이 끊겨도 데이터를 잃지 않는 offline-first 구조를 기본값으로 한다.
5. 위치·장애·복지 관련 데이터는 기능 편의보다 최소수집과 통제를 우선한다.
6. AI는 근거와 불확실성을 드러내며 전문 판단을 대체하지 않는다.
7. 실제 데이터, 합성 데이터, 계획상의 기대 결과를 명확히 분리한다.

## 4. 주요 사용자

| 주체 | 해야 하는 일 | 제품이 제공할 가치 |
| --- | --- | --- |
| 사용자·보호자 | 기기 확인, 주행 기록, 알림 확인 | 낮은 입력 부담, 상태 근거, 점검 시점 |
| 수리사 | QR 확인, 수리·교체 기록, 모델 의견 검토 | 과거 이력, 빠른 기록, 유사 사례와 위험 근거 |
| 복지관 | 사용자·기기·지원·실증 관리 | 기관별 권한, 집계, 보고서, 운영 이상 탐지 |
| 정책 담당자 | 예산·지원 판단 | 개인 원자료가 아닌 검증된 익명 집계 |

## 5. 8개월 내 필수 결과

- 신규 React Native 모바일 앱과 Android/iOS 실기기 검증
- 모바일 자체 GPS 주행 세션과 백그라운드 수집
- SQLite 로컬 이벤트 로그와 멱등 동기화
- 버전이 고정된 텔레메트리·도메인 이벤트 계약
- Firebase Auth/App Check 기반 인증·앱 무결성 검증
- Firestore 기관별 제어 데이터와 Cloud Storage 원본 GPS batch 분리
- 분석 필요 시 BigQuery partition/tenant 격리
- 이벤트 재생 가능한 기기 상태 projection
- 규칙 기반 baseline과 비교되는 데이터 품질 모델
- 수리·주행 이력 기반 신뢰성/생존분석 및 calibration
- 근거 ID를 강제하는 AI 리포트
- 신규 복지관 콘솔과 기관별 문서 adapter
- 관측 가능성, 접근성, 개인정보, 장애복구 검증
- 재현 가능한 최종 평가·데모·기술백서

## 6. 비목표

- 일반 목적 RAG 챗봇
- 포인트·리워드 경제 시스템
- 외부 IoT 센서 또는 과거 GPS API 의존
- 데이터 근거 없는 고장 확정·안전 보증
- 규모를 증명하지 못하는 Kafka/Kubernetes 도입
- 기존 프론트엔드·관리자·백엔드의 점진적 리팩터링

## 7. 성공 판단 기준

### 시스템

- 동일 배치를 반복 전송해도 논리 이벤트가 중복되지 않는다.
- 네트워크 단절과 앱 재시작 후 미전송 이벤트가 복구된다.
- tenant 간 데이터 접근 테스트가 모두 거부된다.
- 원본 위치 삭제 후 운영 집계와 감사 규칙이 정의된 대로 유지된다.

### 모델

- 모든 모델은 규칙 기반 또는 고정주기 baseline과 비교한다.
- 학습·검증·최종 평가는 시간 기준으로 분리한다.
- calibration과 abstention을 별도 지표로 보고한다.
- 데이터가 부족한 부품은 예측 대신 기본 점검주기를 사용한다.

### 보고서

- 주요 주장에는 데이터, 테스트, 모델 실행 또는 사람 검토 근거가 연결된다.
- 실제 근거가 없는 문장은 자동 승인되지 않는다.
- 정기리포트의 계획과 실제 진행이 분리되어 있다.

## 8. 확정된 재사용 경계

- 기존 코드·UI·브랜드: 사용하지 않음
- 기존 구조: 문제·업무 이해를 위한 참고 가능
- 기존 DB 형식·수리데이터: 사용 가능
- 복지관 관계와 실증 협의: 계속 활용 가능
- 이관 구현: 신규 데이터 계약과 품질 게이트를 거친 별도 importer로 한정
