# 2026년 9월 2차 기술 정기리포트

> 문서 상태: 계획 기준 사전작성본  
> 실제 증빙: 회의 후 갱신

## 목적

수리 이력과 누적 주행량을 이용해 부품별 time-to-inspection을 추정할 수 있는지 검증한다. 모델 성능이 충분하지 않으면 규칙 기반 점검 주기를 유지하도록, 실패 가능한 실험으로 정의한다.

## 계획상 논의

- 수리, 실제 고장, 예방 교체, 단순 상담을 outcome에서 구분하는 방법
- 관측 시작·종료, censoring, 부품 교체 후 risk clock 재설정
- 고정 기간, 누적 주행거리 임계값, Cox/Weibull, survival forest 후보
- 동일 기기 이력이 train/test에 섞이지 않는 시간·그룹 분할
- 부품별 표본 부족, 기관·기종 편향, 누락 주행량의 영향
- 승인된 Cloud Storage manifest와 Firestore 수리 export를 결합하고, optional BigQuery 사용 시 partitioned snapshot과 query 비용을 고정하는 데이터 계보
- 사용자에게 고장 예측이 아닌 점검 권장 위험으로 표현하는 원칙

## 결정 예정 항목

- 학습 가능한 outcome과 제외할 수리 유형
- baseline 및 후보 모델의 비교 기준
- 30일 위험도, 권장 점검 구간 등 출력 형식
- 데이터 부족·distribution shift 시 abstention 조건
- 모델 실험·Storage manifest/Firestore export 또는 optional BigQuery snapshot 계보를 보존하는 버전 체계

## 기대 산출물

- 신뢰성 데이터셋 및 모델 카드 초안
- Kaplan–Meier 또는 동등한 부품별 위험 곡선
- 고정 점검주기·거리 규칙·생존모델 비교표
- time-split 평가와 대표 실패 사례
- 부품 건강 화면의 저충실도 설명 UX

## 검증 지표

- outcome과 censoring 생성 규칙이 재현 가능한가
- baseline 대비 discrimination뿐 아니라 calibration도 측정했는가
- 부품별 표본 수와 신뢰구간을 성능표에 함께 표시했는가
- 근거가 부족한 케이스에서 모델이 안전하게 유보하는가
