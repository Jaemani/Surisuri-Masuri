# 2026년 5월 1차 기술 정기리포트

> 문서 상태: 계획 기준 사전작성본  
> 실제 증빙: 회의 후 갱신

## 목적

기존 브랜드·코드 재사용을 전제로 하지 않는 신규 제품의 경계를 정하고, 과거 현장 경험과 사용 가능한 데이터 구조만 요구사항의 근거로 전환한다. 8개월 동안 증명할 기술적 성과와 비범위를 합의한다.

## 계획상 논의

- 신규 제품의 임시 정의: 모바일 GPS와 수리·점검 이력을 결합하는 전동보장구 신뢰성 관리 플랫폼
- 기존 세 시스템은 구조 참고 자료로만 취급하고 신규 코드베이스·브랜드·UI를 구성하는 원칙
- 사용자, 수리사, 복지관 담당자의 핵심 업무와 데이터 접근 경계
- 모바일 수집, Firebase control plane, Go Cloud Run ingest, Cloud Storage telemetry, Firestore 상태 projection, 선택적 BigQuery 분석, ML·근거형 리포트의 시스템 경계
- Kafka·Kubernetes·일반 RAG 챗봇·포인트 기능을 초기 핵심 범위에서 제외하는 이유
- ADR, 제품 업데이트, 심각한 오류, 사람 대상 리포트를 서로 분리해 관리하는 문서 체계

## 결정 예정 항목

- 신규 제품의 한 문장 문제 정의와 성공 조건
- 구조 참고가 허용되는 기존 DB 필드와 신규 스키마에서 폐기할 필드
- 원본 위치, 수리 이력, 개인 식별정보의 소유권과 논리적 분리 원칙
- Android/iOS 동시 지원과 Android 우선 검증 여부
- 8개월 기술 게이트 및 월 2회 보고 기준

## 기대 산출물

- 제품 범위 문서와 비범위 목록
- 컨텍스트 다이어그램 및 8개월 아키텍처 초안
- 기존 데이터 구조 참고 목록과 신규 데이터 사전 초안
- 문서 디렉터리·명명 규칙
- 초기 위험 목록: 위치정보, 데이터 편향, 앱 백그라운드 제약, 예측 과장

## 검증 지표

- 핵심 사용자별 작업 흐름이 최소 1개씩 정의되었는가
- 모든 핵심 데이터에 수집 주체·이용 목적·보존 위치가 연결되었는가
- 기술 게이트마다 검증 가능한 산출물이 정의되었는가
- 비범위 항목이 명시되어 범위 팽창을 통제할 수 있는가
