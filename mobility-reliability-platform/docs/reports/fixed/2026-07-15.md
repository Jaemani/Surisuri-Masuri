## 목적

모바일 앱의 offline GPS batch를 안정적으로 수신하는 Go Cloud Run ingest와 Firebase/GCP 저장 경계를 설계한다. Firebase Auth ID token과 App Check를 검증하고, 원본 압축 batch는 Cloud Storage에, 제한된 receipt와 현재 상태 projection만 Firestore에 두어 운영 편의와 텔레메트리 비용을 함께 통제한다.

## 계획상 논의

- Go Cloud Run ingest가 담당할 Firebase ID token·App Check·batch schema·tenant·idempotency 검증
- request `tenantId`를 후보 scope로만 받고, 검증된 principal과 Firestore membership으로 authorize하며 필요한 배포에서 custom claim을 추가 제한으로 사용하는 방식
- 압축 GPS batch의 Cloud Storage object path, body hash, 크기 제한, immutable 원본 원칙
- Firestore receipt에 batch ID, body hash, object path, 처리 상태, accepted/rejected count, 오류 reason만 저장하는 구조
- GPS sample 하나당 Firestore document write를 만들지 않는 금지 규칙과 Firestore current projection의 제한된 갱신
- Emulator Suite에서 Auth·Firestore·Storage fixture를 사용하고 Security Rules의 tenant·role·prediction 쓰기 차단을 검증하는 방법
- Cloud Run min instances 0, concurrency·memory·timeout과 Storage·Firestore 호출 수를 이용한 batch당 비용 측정

## 결정 예정 항목

- versioned JSON Schema, 압축 형식, 최대 batch 크기와 재전송 계약
- Storage object path의 비식별 규칙과 receipt 스키마
- Firebase Auth·App Check 실패와 membership 불일치의 응답 코드
- Storage 저장과 Firestore receipt 생성 사이의 부분 실패·보상·재처리 방식
- Firestore current projection 갱신 빈도와 sample 수에 독립적인 write 상한
- Emulator/Security Rules 테스트를 배포 게이트로 적용할 범위

- Storage object·Firestore receipt·current projection 스키마와 계보 규칙
- 정상·중복·구버전·권한 오류·부분 실패 계약 테스트
- Emulator fixture와 Firestore/Storage Security Rules 자동 테스트
- 주행 시간·sample 수·batch 크기별 예상 또는 실제 서비스 호출 수와 비용표

## 검증 지표

- 같은 `clientBatchId`와 동일 body를 반복 전송할 때 서버 파생 key가 기존 UUIDv7 batch receipt를 반환하고 object가 중복 생성되지 않는가
- Firebase ID token·App Check·tenant membership 중 하나라도 유효하지 않으면 요청이 거부되는가
- GPS sample 수가 증가해도 Firestore write 수가 sample 수에 비례하지 않는가
- Storage object와 Firestore receipt가 tenant ID, batch ID, body hash로 상호 추적되는가
- Emulator Suite에서 다른 tenant 읽기·쓰기와 client의 raw prediction·role 변경이 차단되는가
- batch당 Cloud Run, Storage, Firestore 사용량과 추정 비용을 동일 fixture로 재현할 수 있는가
