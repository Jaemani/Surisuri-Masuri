## 목적

Cloud Storage 주행 batch와 Firestore의 수리·점검·부품 교체 기록을 시간 순서대로 재생해 Firestore current projection을 재구성하는 과정을 설계한다. 과거 수리 데이터는 출처를 보존한 일회성 이관으로 편입한다.

## 계획상 논의

- Storage batch/receipt에서 유도한 TripRecorded와 Firestore의 RepairLogged, PartReplaced, InspectionCompleted 등 도메인 이벤트
- immutable Storage 원본·Firestore 업무 기록과 서버만 쓸 수 있는 current projection의 역할 분리
- 늦게 도착하거나 정정된 이벤트가 projection에 미치는 영향
- projector 버전 변경 시 승인된 Storage manifest와 Firestore export/snapshot 기반 replay 및 checksum 검증
- 과거 DB 필드의 신규 이벤트 매핑, 미확인 값, 중복 수리 기록 처리
- GPS sample별 Firestore write 없이 trip summary와 현재 상태만 제한된 문서 수로 갱신하는 비용 경계
- Digital Twin 명칭은 시점별 상태 복원이 검증된 이후에만 사용하는 원칙

## 결정 예정 항목

- 도메인 이벤트 목록과 필수 payload
- Firestore 기기·부품 current projection 스키마와 Security Rules
- 정정·취소 이벤트와 원본 불변성 규칙
- migration id, source record id, 변환 버전의 계보 구조
- replay 성능·정합성 검증 방식

## 기대 산출물

- 이벤트 카탈로그와 JSON Schema
- Storage manifest/Firestore snapshot 기반 상태 projector 및 특정 시점 replay PoC
- 기존 수리 데이터 migration dry-run 리포트
- 기기 타임라인과 부품별 상태 화면
- 원본 레코드→이벤트→현재 상태의 계보 조회
- Emulator Suite에서 client의 projection 직접 쓰기를 차단하는 Security Rules 테스트

## 검증 지표

- 같은 이벤트 집합의 반복 replay 결과가 동일한가
- out-of-order 이벤트를 반영한 뒤 상태가 올바르게 교정되는가
- migration 전후 레코드 수와 제외·중복·미확인 수가 설명되는가
- 화면의 상태 값에서 근거 이벤트까지 추적할 수 있는가
- sample 수 증가가 Firestore projection write 수를 비례 증가시키지 않고, client의 projection 직접 쓰기가 차단되는가
