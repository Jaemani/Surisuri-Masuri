## 목적

모바일 수집부터 보고서 생성까지의 운영 상태를 관측하고, 장애·보안·권한 문제를 실제 사용자 영향과 연결해 대응할 수 있는 체계를 검증한다. 실패를 숨기지 않고 탐지·완화·복구·보고의 증거를 남긴다.

## 계획상 논의

- 수집 성공률, offline queue age, Cloud Run 오류율, Storage upload/processing 상태, Firestore receipt·projector lag, 모델 abstention, report validation 실패율
- trace 또는 correlation ID로 모바일 batch→Cloud Storage object/Firestore receipt→current projection→Fact→report 연결
- SLI/SLO 후보와 경보 임계값, 경보 피로 방지
- Cloud Run timeout, Storage 쓰기/읽기 오류, Firestore receipt·projector 오류, Auth/App Check·Security Rules 오설정, 모델 파일 손상, LLM timeout 장애 주입
- Cloud Run·Storage·Firestore와 활성화된 optional BigQuery의 사용량·비용 경보 및 budget alert 검증
- dependency·secret·접근권한·데이터 내보내기 보안 검토
- 큰 글씨·스크린리더·터치 영역·권한 안내를 Android/iPhone 실기기에서 재검증하는 접근성 회귀 점검
- 장애 주입은 `synthetic drill`로 표시하고, 실제 사용자 영향이 incident 기준을 충족할 때만 별도 INC 문서를 여는 경계

## 결정 예정 항목

- pilot에서 반드시 보는 운영 대시보드와 경보
- 심각도 분류 및 사람에게 알릴 조건
- graceful degradation: ML·LLM·지도 계층 장애 시 제공할 기능
- Storage 원본 보존, Firestore export/restore, projector replay의 복구 목표
- 보안 발견 사항의 차단 기준과 조치 기한
- 실기기 접근성 수용 기준과 개선 전후 비교 방법
- synthetic drill 결과를 실제 incident로 승격할 조건

## 기대 산출물

- 운영 대시보드와 trace 예시
- 장애 주입 시나리오 및 synthetic drill timeline
- Storage 원본·Firestore export/restore와 projector replay 검증 결과
- 보안·권한·개인정보 점검표
- Android/iPhone 실기기 접근성 재검증 결과와 개선 전후 화면
- incident 분류 템플릿과 synthetic drill 후속 액션 목록

## 검증 지표

- 주요 장애가 사용자의 신고 전에 시스템에서 탐지되는가
- 특정 report에서 Firestore projection·receipt와 Cloud Storage 원본 batch까지 correlation ID로 추적 가능한가
- ML·LLM 장애가 수집·수리 기록의 핵심 기능을 중단시키지 않는가
- Storage/Firestore 복원 후 누락·중복·projection checksum을 검증했는가
- 보안 발견 사항에 책임자·우선순위·상태가 연결되는가
- 큰 글씨·스크린리더·터치 영역·권한 안내가 두 플랫폼 실기기에서 재검증되었는가
- synthetic drill 산출물이 실제 incident·사용자 영향으로 오인되지 않게 표시되는가
