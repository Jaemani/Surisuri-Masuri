# 리포트 운영 인덱스

이 디렉터리는 8개월 계획을 기준으로 미리 준비한 고정 리포트와 실제 요청에 따라 작성한 사람 대상 리포트를 분리한다. 두 종류 모두 계획·실제·근거를 구분하며, 문서가 존재한다는 사실만으로 회의 수행이나 공식 발행을 뜻하지 않는다.

## 리포트 종류

| 종류 | 위치 | 역할 | 기본 상태 | 발행 조건 |
| --- | --- | --- | --- | --- |
| 고정 리포트 | [`fixed/`](./fixed/) | 2026년 5월~12월 매월 2회, 총 16회의 계획 의제와 예상 산출물 고정 | `draft` 사전작성본 | 실제 세션·진행·결정·증거를 사람이 확인 |
| 요청 리포트 | [`human/`](./human/) | 착수, 구현 점검, 외부 검토 등 요청 시점의 실제 상태 요약 | 자동 초안은 `draft` | 대상 독자, reviewer, `issued_at`, 근거 일치를 사람이 확인 |

## 현재 상태의 해석

- fixed 문서 16개는 5월 15일부터 12월 31일까지의 **승인된 계획 기준선**이다.
- `실제 세션·발행일: 미입력`인 과거 회차는 누락된 성과가 아니라 아직 실제 회의·발행 사실이 확정되지 않은 사전작성본이다.
- human 문서의 `issued_at: TBD`와 `status: draft`는 코드·테스트 근거가 있더라도 공식 외부 발행 전이라는 뜻이다.
- 과거 회차를 발행할 때는 실제 세션이 있었는지 먼저 확인한다. 없었다면 회의를 소급 생성하지 않고, 현재 시점의 별도 상태 리포트를 새 ID로 작성한다.
- 개발 속도가 계획보다 빠르거나 느려도 fixed 문서의 계획을 실제 실적으로 바꾸지 않는다. 차이는 해당 문서의 실제 입력란이나 새 human 리포트에 기록한다.

## 발행 상태 전이

```text
사전 계획 또는 자동 초안(draft)
  -> 실제 정보와 증거 입력
  -> 별도 시점의 사람 검토
  -> 발행 상태와 발행일 확정
  -> 오류 발견 시 정정 또는 superseded 연결
```

Git 커밋, 테스트, 화면 또는 EVD가 있다는 사실은 기술 주장 일부를 뒷받침하지만 실제 참석자·회의 시간·사진·지출을 대신하지 않는다. 외부 행정 양식은 실제 증빙과 해당 양식 규칙을 별도로 따른다.

## 읽는 순서

1. 월별 의제와 날짜는 [fixed 인덱스](./fixed/README.md)에서 확인한다.
2. 실제 구현 상태를 요약한 요청 리포트는 [human 인덱스](./human/README.md)에서 확인한다.
3. 운영 원칙은 [정기보고·회의 운영계획](../plans/REPORTING_AND_MEETING_PLAN.md)을 따른다.
4. 실제 주장에는 [증거 인덱스](../evidence/README.md)를 연결한다.
5. 중요한 결정·제품 변화·심각한 오류는 각각 [ADR](../decisions/), [Product Update](../product-updates/), [Incident](../incidents/)로 분리한다.
