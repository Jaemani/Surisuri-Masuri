- 증명하지 못함: 실제 모바일 payload, 서버 코드, 네트워크 전송, 데이터베이스 constraint, 멱등성 동작.

Stored/rejected/cleanup, released lease, request owner, partial lease 회귀 test를 추가하고 재리뷰에서 잔여 High/Medium 없음을 확인했다.
- WSL2 Docker 전체 build가 repository VCS status를 읽지 못한 실패는 source·test 실패가 아니었다. 동일 source를 `-buildvcs=false`로 build해 executable compile을 확인했다.
- 위 실패는 local 구현·검증 과정이며 production·staging·field runtime, 사용자 데이터 또는 기관 운영에 영향이 없어 Incident 기준에 해당하지 않는다.

### 증명 범위와 한계

- 증명: synthetic current-state 정책과 Firestore Emulator 범위에서 authoritative receipt, current 관계와 fence를 확인한 forward request·opaque grant만 생성한다.
- 증명: claim 이후 current consent withdrawal을 관측하면 새 grant를 만들지 않고 Firestore mutation을 수행하지 않는다.
- 증명하지 못함: 실제 Firebase project의 ADC/IAM, production transaction contention, staging clock, 실제 consent race와 Cloud Storage list/read.
- Firestore authorization transaction과 첫 Storage call 사이의 관계 변경을 완전히 원자화하지 않는다. 30초 이하 TTL, fence/request binding과 provider boundary별 expiry 재검사로 TOCTOU window를 제한한다.

### 관측 결과

- manifest와 raw validator를 독립 method로 분리해 manifest-only와 raw-only 판단에 사용할 수 있다. combined validation도 제공하지만 source bytes나 decoded batch를 결과에 보존하지 않는다.
- manifest validator는 snapshot path·digest·CRC32C·size, exact content headers·`no-store`, writer가 고정한 custom metadata와 제한된 emulator/testbench metadata allowlist를 검증한다.
- manifest bytes는 64KiB strict decoder로 malformed·duplicate·unknown·trailing JSON을 거부한다. receipt request의 tenant, device, trip, installation, batch, client batch, consent revision, body hash, sample count, captured/received/expiry time, validator version과 deterministic raw path를 교차검증한다.

- decoded manifest를 receipt immutable input과 manifest 자체의 bounded raw reference로 다시 생성해 exact canonical bytes를 비교한다. semantic time이 같아도 whitespace, field order와 timezone 표기가 canonical bytes와 다르면 `manifest_noncanonical`로 닫는다.
- raw validator는 gzip 해제 전에 exact compressed bytes의 SHA-256·CRC32C·size와 snapshot·metadata·headers를 비교한다.
- gzip은 단일 stream으로만 읽고 `2MiB+1`까지만 해제한다. corrupt stream, trailing garbage, concatenated stream과 decompressed overflow fixture가 모두 `strict_payload_invalid`로 거부됐다.
- 해제된 body SHA-256은 receipt body hash와 비교하고, strict `telemetry-batch.v2` decoder·validator를 다시 실행한다. tenant, device, trip, installation, consent, client batch, sample count와 captured bounds가 receipt와 다르면 `payload_lineage_mismatch`다.
- 현재 `TelemetryValidatorVersion`만 explicit registry에 등록한다. duplicate version은 malformed/valid 등록 순서와 무관하게 전부 비활성화하고 profile의 decompression 상한은 global 2MiB를 넘을 수 없다.
- codec profile은 synthetic canonical raw fixture와 literal compressed SHA-256 golden으로 startup self-test한다. toolchain·compressor output 또는 valid raw의 recompression이 profile과 다르면 content conflict로 추정하지 않고 `codec_profile_unavailable`로 닫는다.
- combined reason은 ADR precedence에 따라 `unavailable -> metadata -> manifest -> raw`로 합성한다. raw header 손상과 manifest raw-reference mismatch가 동시에 있는 재현 fixture에서 metadata reason이 우선했다.
- 최초 독립 리뷰에서 발견된 accepted snapshot reason 비대칭, 동시 손상 precedence 역전, duplicate registry의 순서 의존성을 수정하고 regression test로 고정했다.

### 증명 범위와 한계

- 증명: synthetic manifest·raw·snapshot에서 canonicality, receipt cross-lineage, compressed/decompressed integrity, strict payload, gzip bound, registry·codec self-test와 bounded reason precedence가 코드 계약대로 동작한다.
- 증명하지 못함: 실제 GCS reader가 반환한 bytes와 content validator를 하나의 orchestration에서 연결한 end-to-end 분류, version·soft-delete inventory, pre/post read drift와 열 classification matrix.

- reflection test는 raw content result가 source slice를 담지 않는 shape만 확인한다. final classifier result·metrics·structured log 전체의 개인정보 scan을 완료했다는 증거가 아니다.
- actual GCS IAM·lifecycle·staging, Cloud Run, 실제 사용자 GPS와 수리데이터는 검증하지 않았다.
- 이 pure validator는 object create/delete, receipt/index mutation, recovery attempt completion, cleanup 또는 purge를 수행하지 않는다.

### 관측 결과

- production factory는 `storage.NewClient`와 read-only OAuth scope를 직접 소유하고 임의 client option이나 외부 `BucketHandle`을 받지 않는다. `NewGRPCClient` 경로를 노출하지 않아 raw compressed-byte read가 필요한 adapter를 HTTP client 경계에 고정한다.
- exact object path마다 두 query를 분리한다.
  - non-soft-deleted inventory: `Versions:true`, `SoftDeleted:false`
  - soft-deleted inventory: `Versions:false`, `SoftDeleted:true`
- 두 query 모두 `Prefix`와 `MatchGlob`을 exact path로 설정하고 `ProjectionNoACL`을 사용한다. 반환 attrs의 name이 exact path와 다르거나 generation·metageneration·size, soft-delete marker와 duplicate generation 구조가 기대와 다르면 coverage를 complete로 만들지 않고 `response_unverifiable`로 fail-closed한다.
- inventory caller limit은 2~100으로 제한한다. 각 iterator의 page max를 `limit+1`로 두고 실제 candidate도 `limit+1`번째까지만 관찰한 뒤, 반환 candidate는 limit개로 제한하고 `truncated=true`를 남긴다. 하나를 자동 선택하기 위해 더 읽지 않는다.
- manifest와 raw read는 exact object generation에 더해 `GenerationMatch`와 `MetagenerationMatch` 조건을 함께 사용한다. provider의 412/failed-precondition은 generation missing이 아니라 `precondition_drift`로 정규화한다.
- manifest read는 `ReadCompressed(false)`, raw generation read는 `ReadCompressed(true)`를 고정한다. 두 read 모두 `maxBytes+1`까지만 읽고 상한을 넘으면 content를 반환하지 않은 채 typed `read_limit_exceeded`로 닫는다.

- provider detail을 노출하지 않는 typed error로 permission denied, quota limited, provider timeout, provider cancelled, provider unavailable, response unverifiable, exact generation not found, precondition drift와 read limit exceeded를 구분한다. caller context가 이미 cancel/deadline 상태면 표준 context error를 보존한다.
- object snapshot에는 path, SHA-256 metadata, CRC32C, size, generation, metageneration, content type/encoding, cache control과 cloned metadata만 담는다. provider attrs 또는 result 쪽 map 변경이 서로 전파되지 않는 synthetic test가 통과했다.

### 관측 결과

- `RecoveryLeaseStore`와 `CleanupTransitionStore` 경계를 추가하고 `ClaimRecoveryLease`, `RenewLease`, `BeginCleanupTransition`을 Firestore transaction으로 구현했다.
- `RecoveryAttemptProposal`은 UUID attempt ID와 server allowlist의 exact worker version을 요구한다. 현재 허용값은 `telemetry-recovery.v1`이며 이메일·UID·App ID와 임의 버전 문자열은 거부한다.
- 최초 request admission은 recovery attempt가 아니므로 nested attempt document와 `recovery_attempt_count`를 만들지 않는다.
- 만료된 request replay takeover와 explicit sweeper recovery claim은 한 transaction에서 다음 작업을 함께 수행한다.
  - 3-way index/receipt linkage와 `reserved` 상태 검증
  - fencing token과 receipt revision 각각 1 증가
  - `recovery_attempt_count` 1 증가
  - 새 lease owner·heartbeat·expiry 기록
  - `/ingestReceipts/{receiptId}/recoveryAttempts/{attemptId}`에 `started` attempt 생성
- `started` attempt에는 attempt·tenant·receipt ID, owner kind, fencing token, worker version과 시작시각만 기록한다. telemetry body, 좌표, UID, App ID, 인증 토큰은 기록하지 않는다.
- `ClaimRecoveryLease`는 sweeper owner만 허용한다. active lease는 `lease_held`, safe-release backoff 전에는 `lease_not_due`, deadline 이후에는 `deadline_elapsed`, terminal receipt에는 `not_eligible`을 read-only로 반환하고 receipt update나 attempt 생성을 하지 않는다.
- 동시 sweeper claim은 `lease_acquired` 1건과 `lease_held` 1건으로 수렴했다. winning receipt만 token·revision 2, attempt count 1이 됐고 `started` attempt도 1개만 생성됐다.
- request-side expired takeover도 경쟁 request 중 하나만 lease를 얻고 다른 request는 in-progress로 수렴했다. winning request의 owner kind와 fencing token을 가진 `started` attempt만 생성됐다.
- HTTP replay는 authorization/receipt read time의 coherence를 확인하고 더 늦은 receipt server time으로 같은 authorization snapshot을 다시 평가한다. 최초 평가 뒤 receipt read 전에 consent·trip이 시간 만료된 fixture는 update와 attempt 생성 0으로 닫혔다.
- 동일 attempt ID 문서가 이미 있는 sweeper claim은 nested `Create` 실패와 receipt update가 함께 rollback됐다. token·revision·attempt count·owner·lease timestamp와 기존 attempt document가 모두 불변이었다.
- `RenewLease`는 동일 owner·token·exact expiry를 가진 현재 fence가 만료 전 45초 renewal window에 들어왔을 때만 허용한다.
- renewal은 fencing token을 바꾸지 않고 heartbeat, lease expiry, `next_recovery_at`과 receipt revision만 갱신한다. 새 expiry는 기존 expiry보다 늦고 reservation deadline 이하여야 한다.
- 너무 이른 renewal, expiry 시각의 renewal, stale owner·token과 deadline을 넘는 연장은 update 0으로 fail-closed한다.
- renew와 takeover 경쟁은 transaction commit 순서에 따라 둘 중 하나만 이긴다. renew가 이기면 기존 token 1과 owner가 유지되고 claim은 held가 된다. takeover가 이기면 token 2와 새 owner가 남고 stale renew는 거부된다. 두 경우 모두 receipt revision은 한 번만 증가한다.
- `BeginCleanupTransition`은 3-way linkage가 정상인 `reserved` receipt에만 적용한다. `cleanupAt=min(application UTC, receipt snapshot read time)`을 사용해 어느 clock이라도 deadline 전이면 조기 cleanup을 시작하지 않는다.
- deadline이 지났고 active lease가 없으면 하나의 transaction에서 token·revision을 각각 1 증가시키고 lease를 제거한 뒤 다음 값을 고정한다.
  - `status=cleanup_pending`
  - `cleanup_mode=reservation_expiry`
  - `cleanup_origin_status=reserved`
  - `cleanup_quiescence_until=max(cleanupAt, last lease expiry)+6분`
- 이미 같은 reserved-origin cleanup transition이 시작된 경우 `transition_already_started`, deadline 전이나 active lease 상태는 `transition_not_ready`, 다른 상태는 `transition_not_eligible`을 update 없이 반환한다.
- deadline의 cleanup transition과 recovery claim이 경쟁하면 cleanup transition만 시작되고 recovery claim은 `deadline_elapsed` 또는 `not_eligible`로 닫힌다.
- cleanup transition과 stale `MarkStored`·`MarkRejected` 경쟁에서는 `cleanup_pending` 상태와 증가한 token이 남고 stale finalizer는 terminal field를 기록하지 못한다.
- `MarkStored`·`MarkRejected`·`ReleaseLease`는 receipt revision 정수 상한을 넘기지 않고 update 0으로 fail-closed한다. year 1/year 9999 극단 clock delta도 duration overflow 없이 admission과 cleanup 양쪽에서 거부됐다.
- reserved-origin `cleanup_pending`과 `expired`는 stored artifact field와 sample이 없어야 한다. pending은 transition 시각보다 quiet-period가 뒤여야 하고, expired completion 시각은 quiet-period 이전일 수 없도록 상태 불변조건을 분리했다.
- targeted unit test와 Firestore Emulator integration entrypoint 8개가 최종 통과했다.
  - concurrent same-batch admission
  - missing authorization write-zero
  - concurrent expired request takeover
  - expired owner 대 takeover finalizer
  - concurrent sweeper claim
  - duplicate attempt create의 claim transaction 전체 rollback
  - renew 대 takeover
  - cleanup transition 대 deadline recovery/finalizer

Official Storage testbench와 container smoke는 `.github/workflows/ci.yml`의 고정 image/digest 및 fail-closed 검사와 같은 절차로 실행했다.

### 관측 결과

- `forward_recovery`와 `accepted_integrity_audit`만 허용하고 각 purpose의 receipt state, lineage와 fence 제약을 terminal 결과에서도 다시 검사한다.
- grant·request binding과 caller context를 모든 list/inspect/read 직전에 재검증한다. provider context deadline은 caller, grant expiry와 forward fence expiry 중 가장 이른 시각이며 authorization 제한으로 끝난 timeout은 provider timeout이 아니라 `ErrArtifactReadAuthorizationExpired`다.
- manifest/raw inventory는 각각 regular version과 soft-deleted query가 모두 수행된 complete coverage를 요구하고 limit 2를 사용한다. 복수 candidate, 잘못된 exact path, 중복 generation, truncated-under-bound와 예상 밖 soft-delete를 자동선택하지 않는다.
- unique candidate는 pre-inspect, exact generation+metageneration bounded read, post-inspect를 거친다. direct NotFound는 `generation_changed_during_read`, precondition drift는 `metageneration_changed_during_read`, 동일 lineage에서 attrs가 바뀐 응답은 `response_unverifiable`로 닫힌다.
- forward는 manifest가 없을 때만 deterministic raw-only를 탐색한다. valid manifest가 있으면 그 manifest가 strict validation 후 가리킨 exact raw generation 외 fallback을 사용하지 않는다.
- accepted audit은 receipt가 고정한 raw·manifest generation만 권위값으로 사용한다. exact accepted generation이 모두 없으면 `stored_missing`, 다른 generation이 있으면 `generation_drift`, exact soft-deleted generation은 bounded stored-missing reason으로 구분한다.
- 두 accepted artifact가 모두 존재할 때 stable read를 먼저 끝낸 뒤 combined validator를 호출해 unavailable→metadata→manifest→raw content precedence를 지킨다.
- 열 classification과 purpose별 허용 reason matrix 26 case가 통과했다. unknown purpose·classification·reason과 forward-only/accepted-only 결과는 fail-closed한다.
- provider permission·quota·timeout·cancel·unknown error와 incomplete inventory는 missing으로 강등하지 않는다.
- request의 `ForwardFence`, accepted raw lineage와 accepted manifest lineage는 진입 시 복제해 provider callback 사이 caller-side 변경이 내부 authorization binding을 바꾸지 못한다.
- 결과는 classification, low-cardinality reason, inventory count/coverage, validator version, observed time과 digest/generation pin만 담는다. value scan에서 raw body, object path, body hash, tenant identity가 결과에 남지 않았다.
- classifier는 reader와 pure validator 외 write port를 받지 않으며 object create/delete, receipt/index/attempt mutation을 수행하지 않는다.
- 독립 리뷰에서 unknown-purpose outcome과 accepted 조기 content precedence 문제를 수정했다. 회귀 테스트 뒤 잔여 High/Medium이 없음을 재확인했다.
- 최종 WSL2 local에서 전체 Go gofmt/module verify/vet/race/build와 workspace check/build/test가 통과했다.
  - Markdown local links: 93개
  - mobile policy: 65 tests
  - Firebase Rules: 24 tests
  - contracts: v2·v1 compatibility·domain event valid/invalid 6 fixtures
  - Android·iOS Expo static export: 양쪽 통과
- pinned official Storage testbench에서 exact-generation reader와 immutable store/replay integration 2건이 race detector와 함께 통과했다.
- local container image `sha256:678f3db85dd9541f6181c7c246b2467683fe410b09bcc69f74e36143dd26eda4`를 만들고 `/healthz=200`, `/readyz=503`, ingest `503`을 확인한 뒤 smoke container를 제거했다.
- 동일 source commit의 GitHub clean runner [29848650699](https://github.com/Jaemani/Surisuri-Masuri/actions/runs/29848650699)이 5분 14초에 성공했다. workspace, Firebase Emulator, Firestore admission race, pinned Storage testbench, Go race/vet/module, container build와 fail-closed smoke가 모두 통과했다.

### 검증 과정에서 확인한 실패와 정정

- 신규 outcome matrix는 unknown purpose가 `valid_complete`를 허용하는 helper-level fail-open을 발견했다. public `Classify`의 앞선 request validation에만 의존하지 않고 outcome helper 자체가 valid purpose를 먼저 검사하도록 수정했으며 26-case matrix를 다시 통과했다.
- 독립 리뷰는 accepted flow가 manifest content conflict에서 조기 종료해 raw metadata conflict 또는 provider unavailable을 관찰하지 못하는 precedence 문제를 발견했다. 두 stable read 뒤 combined validation을 수행하도록 수정하고 동시 손상·permission/timeout 회귀 테스트를 추가했다.
- 첫 전체 Docker gate는 gateway 하위 디렉터리만 mount해 상위 contract fixture를 찾지 못했다. production code를 바꾸지 않고 platform root 전체를 CI와 같은 경로로 mount해 전체 gate를 다시 통과했다.
- official testbench의 첫 targeted 명령은 shell regex를 과도하게 escape해 `no tests to run`으로 끝났다. 이를 성공 증거로 사용하지 않고 CI와 같은 `Artifact(Store|Reader)Emulator` pattern으로 재실행해 두 integration test의 실제 실행·통과를 확인했다.
- 위 실패는 local 구현·검증 과정이며 production·staging·field runtime, 사용자 데이터 또는 기관 운영에 영향이 없어 Incident 기준에 해당하지 않는다.

### 증명 범위와 한계

- 증명: local synthetic artifact와 official Storage testbench reader 범위에서 request/grant binding, inventory ambiguity, pinned read drift, strict content와 열 classification이 정한 결과를 side effect 없이 만든다.
- 증명하지 못함: current tenant·installation·trip·assignment·precise-location consent를 재평가해 production grant를 발급하는 authorizer.
- classifier constructor는 package-private이며 startup executable, HTTP handler, worker 또는 scheduler에 연결하지 않았다. runtime readiness와 ingest는 계속 `503`이다.
- official testbench는 HTTP GCS reader/store의 exact generation 경계를 검증한다. classifier orchestration 자체는 provider-neutral scripted fake를 사용했으며 staging GCS end-to-end 증거가 아니다.
- actual GCS bucket의 versioning·soft-delete·lifecycle·retention·IAM list 범위, ADC와 비용은 staging에서 검증하지 않았다.
- R6 forward reconciler, manifest conditional create, fenced finalizer, recovery attempt completion, sweeper pagination/backoff, hold·cleanup·purge는 구현하지 않았다.
- result value shape는 검사했지만 향후 runtime metrics·structured log·attempt document의 end-to-end privacy scan은 남아 있다.
- 실제 사용자 GPS, 복지관 현장, 성능·비용·신뢰성 지표를 검증하지 않았다.
