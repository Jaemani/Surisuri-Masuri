- 증명하지 못함: 실제 모바일 payload, 서버 코드, 네트워크 전송, 데이터베이스 constraint, 멱등성 동작.

### 관측 결과

- Firestore candidate adapter는 `status == reserved`와 `next_recovery_at <= scan_cutoff`을 함께 적용하고 `(next_recovery_at, document ID)` 순서의 deterministic cursor와 `limit+1`로 다음 페이지 존재 여부를 결정한다. 한 run에서 고정한 cutoff는 checkpoint와 함께 보존해 cutoff 뒤 새로 due가 되는 정상 유입이 현재 epoch tail을 계속 늘리는 것을 제한한다. 이는 read snapshot 고정이 아니다.

### 관측 결과

- manifest와 raw validator를 독립 method로 분리해 manifest-only와 raw-only 판단에 사용할 수 있다. combined validation도 제공하지만 source bytes나 decoded batch를 결과에 보존하지 않는다.

- raw validator는 gzip 해제 전에 exact compressed bytes의 SHA-256·CRC32C·size와 snapshot·metadata·headers를 비교한다.
- gzip은 단일 stream으로만 읽고 `2MiB+1`까지만 해제한다. corrupt stream, trailing garbage, concatenated stream과 decompressed overflow fixture가 모두 `strict_payload_invalid`로 거부됐다.

- codec profile은 synthetic canonical raw fixture와 literal compressed SHA-256 golden으로 startup self-test한다. toolchain·compressor output 또는 valid raw의 recompression이 profile과 다르면 content conflict로 추정하지 않고 `codec_profile_unavailable`로 닫는다.
- combined reason은 ADR precedence에 따라 `unavailable -> metadata -> manifest -> raw`로 합성한다. raw header 손상과 manifest raw-reference mismatch가 동시에 있는 재현 fixture에서 metadata reason이 우선했다.
- 최초 독립 리뷰에서 발견된 accepted snapshot reason 비대칭, 동시 손상 precedence 역전, duplicate registry의 순서 의존성을 수정하고 regression test로 고정했다.

### 증명 범위와 한계

- 증명: synthetic manifest·raw·snapshot에서 canonicality, receipt cross-lineage, compressed/decompressed integrity, strict payload, gzip bound, registry·codec self-test와 bounded reason precedence가 코드 계약대로 동작한다.
- 증명하지 못함: 실제 GCS reader가 반환한 bytes와 content validator를 하나의 orchestration에서 연결한 end-to-end 분류, version·soft-delete inventory, pre/post read drift와 열 classification matrix.

- actual GCS IAM·lifecycle·staging, Cloud Run, 실제 사용자 GPS와 수리데이터는 검증하지 않았다.
- 이 pure validator는 object create/delete, receipt/index mutation, recovery attempt completion, cleanup 또는 purge를 수행하지 않는다.

- request의 `ForwardFence`, accepted raw lineage와 accepted manifest lineage는 진입 시 복제해 provider callback 사이 caller-side 변경이 내부 authorization binding을 바꾸지 못한다.

- classifier는 reader와 pure validator 외 write port를 받지 않으며 object create/delete, receipt/index/attempt mutation을 수행하지 않는다.
- 독립 리뷰에서 unknown-purpose outcome과 accepted 조기 content precedence 문제를 수정했다. 회귀 테스트 뒤 잔여 High/Medium이 없음을 재확인했다.
- 최종 WSL2 local에서 전체 Go gofmt/module verify/vet/race/build와 workspace check/build/test가 통과했다.

  - mobile policy: 65 tests

  - Android·iOS Expo static export: 양쪽 통과

### 검증 과정에서 확인한 실패와 정정

- 신규 outcome matrix는 unknown purpose가 `valid_complete`를 허용하는 helper-level fail-open을 발견했다. public `Classify`의 앞선 request validation에만 의존하지 않고 outcome helper 자체가 valid purpose를 먼저 검사하도록 수정했으며 26-case matrix를 다시 통과했다.
- 독립 리뷰는 accepted flow가 manifest content conflict에서 조기 종료해 raw metadata conflict 또는 provider unavailable을 관찰하지 못하는 precedence 문제를 발견했다. 두 stable read 뒤 combined validation을 수행하도록 수정하고 동시 손상·permission/timeout 회귀 테스트를 추가했다.
- 첫 전체 Docker gate는 gateway 하위 디렉터리만 mount해 상위 contract fixture를 찾지 못했다. production code를 바꾸지 않고 platform root 전체를 CI와 같은 경로로 mount해 전체 gate를 다시 통과했다.
- official testbench의 첫 targeted 명령은 shell regex를 과도하게 escape해 `no tests to run`으로 끝났다. 이를 성공 증거로 사용하지 않고 CI와 같은 `Artifact(Store|Reader)Emulator` pattern으로 재실행해 두 integration test의 실제 실행·통과를 확인했다.
- 위 실패는 local 구현·검증 과정이며 production·staging·field runtime, 사용자 데이터 또는 기관 운영에 영향이 없어 Incident 기준에 해당하지 않는다.

### 증명 범위와 한계

- 증명: local synthetic artifact와 official Storage testbench reader 범위에서 request/grant binding, inventory ambiguity, pinned read drift, strict content와 열 classification이 정한 결과를 side effect 없이 만든다.
- 증명하지 못함: current tenant·installation·trip·assignment·precise-location consent를 재평가해 production grant를 발급하는 authorizer.
- classifier constructor는 package-private이며 startup executable, HTTP handler, worker 또는 scheduler에 연결하지 않았다. runtime readiness와 ingest는 계속 `503`이다.
- official testbench는 HTTP GCS reader/store의 exact generation 경계를 검증한다. classifier orchestration 자체는 provider-neutral scripted fake를 사용했으며 staging GCS end-to-end 증거가 아니다.
- actual GCS bucket의 versioning·soft-delete·lifecycle·retention·IAM list 범위, ADC와 비용은 staging에서 검증하지 않았다.
- R6 forward reconciler, manifest conditional create, fenced finalizer, recovery attempt completion, sweeper pagination/backoff, hold·cleanup·purge는 구현하지 않았다.
- result value shape는 검사했지만 향후 runtime metrics·structured log·attempt document의 end-to-end privacy scan은 남아 있다.
- 실제 사용자 GPS, 복지관 현장, 성능·비용·신뢰성 지표를 검증하지 않았다.
