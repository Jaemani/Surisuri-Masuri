PASS valid domain event

첫 실행에서는 동일한 `$id`를 가진 schema를 case마다 다시 compile해 Ajv가 두 번째 case에서 중복 schema 오류를 냈다. schema path별 validator cache를 추가한 뒤 전체 fixture를 다시 실행해 위 결과를 확인했다. 이 오류는 런타임 장애가 아니라 로컬 계약 검사기의 최초 구현 오류이며, 실패 실행을 성공 결과로 숨기지 않는다.

- 증명하지 못함: 실제 모바일 payload, 서버 코드, 네트워크 전송, 데이터베이스 constraint, 멱등성 동작.

## EVD-20260721-013 — Firestore client 최소권한 read matrix

- 생성: 2026-07-21 KST
- 생성자: Codex와 delegated Rules implementation/review
- 환경·데이터: WSL2 local / Firebase Emulator demo project / synthetic tenant·membership·domain documents
- 상태: `generated` — local Emulator 검증 완료, push 후 clean CI와 staging 배포 전
- 대상:
  - `firestore.rules`
  - `packages/firebase-rules/test/firestore.rules.test.js`
  - `packages/firebase-rules/README.md`
  - `docs/decisions/ADR-0014-firestore-client-read-boundary.md`

### 확인 명령

```bash
rtk pnpm check:firebase
rtk pnpm check
rtk pnpm test
```

### 관측 결과

- Firestore Rules Emulator: Firestore 22건과 Storage 2건, 총 24건 통과
- active tenant의 path/field `tenant_id`, active membership의 path/field UID·tenant·roles·유효기간을 모두 확인
- beneficiary: own people 단건 get과 own relationship·assignment·trip·consent·alert get, `tenant_id + person_id` 제한 query 허용; people list·other-person·unfiltered query 거절
- `case_worker`, `tenant_admin`: tenant field가 일치하는 운영 projection read 허용
- guardian·repairer·auditor: own person record 외 other-person·device 등 operational direct read 거절
- suspended/mismatched tenant, cross-tenant, unknown role, tenant/membership list, 모든 client mutation, current consent/ingest internal read/write 거절
- top-level domain document의 stored `tenant_id`가 path tenant와 다르거나 people path ID와 stored `person_id`가 다르면 owner·staff 모두 거절

### 검증 절차 중 실패와 정정

- top-level resource의 path/field tenant 일치 검사를 추가한 첫 실행에서 기존 list query 두 개가 `tenant_id` constraint를 증명하지 못해 24건 중 2건이 실패했다.
- Rules를 느슨하게 되돌리지 않고 beneficiary query에는 `tenant_id + person_id`, staff query에는 `tenant_id` filter를 명시했다. 재실행에서 24건 모두 통과했다.
- people list에 path/field person ID 일치까지 요구하는 추가 실험은 collection query에서 document ID 조건을 증명하지 못해 1건 실패했다. people은 앱이 자신의 known person ID를 단건 get하는 계약으로 더 좁히고 list를 명시적으로 거절했다.
- 이 실패는 local contract test에서 발견됐고 production·field 배포나 사용자 데이터 영향은 없어 Incident 기준에 해당하지 않는다.

### 증명 범위와 한계

- 증명: local Emulator가 평가한 synthetic get/list에서 active membership 하나만으로 다른 person 또는 운영 projection을 읽던 기존 경계를 owner/staff matrix로 좁혔다.
- 증명: query에 필요한 tenant/person constraint와 server-only/write deny가 Rules regression으로 고정됐다.
- 증명하지 못함: production Firebase Rules 배포, 실제 composite index, 실제 mobile/admin query 전체, Admin SDK backend authorization, guardian relationship DTO와 repair data access grant.
- Firebase Admin SDK는 Rules를 우회하므로 이 결과가 backend read/write의 최소권한을 증명하지 않는다.

## EVD-20260721-015 — Firestore admission transaction Emulator integration

- 생성: 2026-07-21 KST
- 생성자: Codex
- 환경·데이터: WSL2 local / Firebase Firestore Emulator / Docker Go 1.26.5 bookworm / synthetic control documents
- 상태: `generated` — local Emulator integration 통과, push 후 clean CI 전
- 대상:
  - `services/telemetry-gateway/internal/firebaseadapter/firestore_admission_integration_test.go`
  - `.github/workflows/ci.yml`
  - `docs/development/WSL_RUNBOOK.md`

### 확인 명령

```bash
rtk pnpm --filter @mobility-reliability/firebase-rules exec firebase emulators:exec \
  --config ../../firebase.json \
  --project demo-mobility-reliability \
  --only firestore \
  "rtk docker run --rm --network host \
    -e FIRESTORE_EMULATOR_HOST=127.0.0.1:8080 \
    -v mobility-go-mod-cache:/go/pkg/mod \
    -v mobility-go-build-cache:/root/.cache/go-build \
    -v /home/jaeman/Codes/Surisuri-Masuri/mobility-reliability-platform:/workspace:ro \
    -w /workspace/services/telemetry-gateway \
    golang:1.26.5-bookworm \
    go test -mod=readonly -count=1 -race -timeout 60s ./internal/firebaseadapter \
      -run FirestoreAdmissionStoreEmulator -v"
```

### 관측 결과

- 실제 Firestore Go client와 Emulator transaction을 사용한 integration entrypoint 2개가 race detector와 함께 통과했다.
- 동일 tenant·installation·client batch·body hash에 서로 다른 proposed server receipt ID 두 개를 동시에 제출했다. test-only transaction decorator는 두 첫 callback이 client-batch index 부재를 모두 읽은 뒤 같은 barrier에서 만나도록 강제했다. authorization load가 3회 이상 실행돼 commit 충돌 뒤 callback retry가 있었음을 확인했다.
- 결과는 `created` 1건과 `replay_pending` 1건이었고 두 응답은 같은 winning receipt ID·batch ID를 반환했다.
- `ingestIdempotency`, `ingestClientBatches`, `ingestReceipts`에는 각각 문서 한 개만 남았다. winning receipt를 실제 Firestore `DataTo`로 decode한 결과와 반환 receipt lineage가 일치했다.
- 두 index를 각각 decode해 같은 winning lineage임을 확인했고 losing proposed receipt path는 `NotFound`여서 충돌 시도 document가 commit되지 않았음을 확인했다.
- current precise-location consent state를 삭제한 synthetic fixture는 `batch_unauthorized`로 끝났고 세 admission collection의 document count가 모두 0이었다.
- integration case는 `FIRESTORE_EMULATOR_HOST`가 있을 때만 실행하며 일반 readonly Go test에서는 skip한다.
- CI에 일반 workspace test와 별도인 `Test Firestore admission transactions` gate를 추가했다. Firebase test를 같은 workspace에서 병렬 실행하지 않는다.

### 증명 범위와 한계

- 증명: local Firestore SDK와 Emulator가 실제 transaction commit/retry를 수행할 때 concurrent same-batch가 한 3-way set으로 직렬화되고 missing authorization에서 write를 만들지 않는다.
- 증명하지 못함: production Firestore, ADC/IAM, 네트워크 단절·deadline, 부하, 실제 consent producer transaction과 admission의 철회 경쟁, partial/corrupt persisted fixture.
- Admin Go client의 Emulator 접근이므로 Firebase Security Rules나 App Check를 검증하는 증거가 아니다. Rules는 EVD-013, token verifier는 EVD-011의 별도 경계다.
- 원본 GPS, 실제 UID/App ID, 사용자·기관 데이터, registry push, Cloud Run 배포는 포함하지 않았다.
