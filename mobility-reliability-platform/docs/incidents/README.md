# 심각한 오류·인시던트 기록

이 디렉터리는 사용자 안전, 위치정보 보호, 데이터 무결성 또는 핵심 운영에 영향을 준 **심각한 실패**와 그 사후분석을 보관한다. 일반 버그 목록이 아니다.

## Severity 기준

| 등급 | 기준 | 예시 | 초기 대응 목표 | 사후분석 |
| --- | --- | --- | --- | --- |
| `SEV-0` | 안전·개인정보·데이터에 치명적이거나 회복 불가능한 영향 | 권한 없는 원본 위치 노출, 광범위한 영구 데이터 손실, 잘못된 점검 안내가 즉각적 안전 위험을 만듦 | 발견 즉시 완화, 필요 시 수집·배포 중단 | 필수, 안정화 후 3영업일 이내 초안 |
| `SEV-1` | 핵심 기능의 광범위한 중단 또는 복구 가능한 중대한 데이터 오류 | 다수 사용자의 GPS 이벤트 유실·중복, 수리 기록 조회 불가, 기관 간 데이터 노출 가능성 | 당일 완화 착수 | 필수, 안정화 후 5영업일 이내 초안 |
| `SEV-2` | 제한된 사용자·환경의 중대한 성능 저하, 우회 가능 | 특정 OS 버전에서 장시간 백그라운드 수집 중단, 일부 보고서 계산 오류 | 다음 업무 주기 내 분류·완화 | 1시간 이상 지속, 반복 발생 또는 데이터 수정이 필요할 때 작성 |
| `SEV-3` | 경미한 결함, 외관 문제, 단발성 개발 오류 | 문구 오탈자, 개발 환경에서만 재현되는 경고 | 일반 백로그 | 이 디렉터리에 작성하지 않음 |

심각도를 확정하기 전에는 상향 가정하고 대응할 수 있다. 낮추거나 높였으면 시각과 근거를 타임라인에 남긴다.

## 작성 트리거와 수명주기

1. SEV-0/1 또는 조건을 충족한 SEV-2를 발견하면 즉시 문서를 만들고 상태를 `investigating`으로 둔다.
2. 확산을 멈추는 동안 확인된 사실만 타임라인에 추가한다.
3. 사용자 영향이 멈추면 `mitigated`, 정상 서비스와 데이터 정합성이 확인되면 `resolved`로 바꾼다.
4. 근본원인, 기여 요인, 탐지 실패, 시정·예방 조치를 포함한 사후분석을 완료하면 `closed`로 바꾼다.
5. 후속 조치는 담당자·기한·검증 증거를 가져야 하며, 새로운 구조 선택이 필요하면 ADR을 작성한다.

## 파일과 필수 필드

- 파일명: `YYYY-MM-DD-sevN-short-slug.md`
- ID: `INC-YYYYMMDD-NN`
- 필수: 심각도, 상태, 탐지·시작·완화·복구 시각, 영향 범위, 탐지 경로, 사실 기반 타임라인, 근본원인, 기여 요인, 대응과 복구, 데이터 수정, 재발 방지, 검증 근거, 관련 링크
- 모든 시각은 `YYYY-MM-DD HH:mm KST` 형식을 사용하며, 모르면 `unknown`으로 명시한다.

## 사후분석 원칙

- 개인 비난이 아니라 시스템 조건과 방어선 실패를 분석한다.
- 직접 원인, 근본원인, 기여 요인, 아직 모르는 항목을 분리한다.
- 사용자가 실제로 받은 영향과 최악의 가능성을 구분한다.
- 복구가 데이터 삭제·재처리를 포함하면 대상, 승인, 검증, 복구 가능성을 기록한다.
- `사람이 조심한다`만을 재발 방지책으로 두지 않는다. 자동 테스트, 관측, 권한, 데이터 계약 또는 런북 개선으로 연결한다.

## 금지사항

- 원인이 확정되기 전에 추측을 사실로 기록하지 않는다.
- 이용자 이름, 연락처, 원본 GPS 좌표, 토큰, 키, 보안 취약점의 악용 세부를 남기지 않는다.
- 심각도를 낮춰 문서화를 피하거나, 여러 관련 사고를 하나의 경미한 버그로 축소하지 않는다.
- 인시던트 문서를 제품 업데이트나 회의록으로 대체하지 않는다.
